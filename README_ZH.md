# Agent Novel Skill — 中文详细文档

> Claude Code 小说创作技能。一个文件。零依赖。78 章连载实战踩坑提炼。

## 这是什么

Agent Novel Skill 是一份 Claude Code 的 skill 定义文件。放到 `~/.claude/skills/agent-novel/` 目录下，在 Claude Code 中输入 "写第X章"，skill 自动加载——6 条硬性规则 + 6 步流水线 + 5 层质检 + L1 冻结摘要，帮你完成从规划到质检的完整一章写作流程。

**核心定位**：不是 AI 代笔。AI 写初稿，你做决定。这个 skill 的作用是——让 AI 写的初稿不那么"AI"，让你做决定时有数据支撑。

## 为什么你需要这个 Skill

### 问题：AI 写小说有三大痛点

1. **AI 腔** — "他感到一阵恐惧""仿佛整个世界都在崩塌""不是在逃避，而是在面对"。读者翻三页就弃了。
2. **低级错误** — 正文出现"第X章"、"四个字"实际五个字、5座车塞8个人。作者自己很难逐条发现。
3. **长篇失控** — 写到第 80 章，忘了第 20 章埋的伏笔。读者记得，你忘了。

### 解决方案

| 痛点 | Agent Novel Skill 的应对 |
|------|--------------------------|
| AI 腔 | 6条硬性规则——不靠审美判断，靠规则执行。抽象情绪词禁止、"不是X是Y"限2次、段落250字上限、句式不连续复用。 |
| 低级错误 | 5层质检——第1层脚本自动扫描(2秒)，第2-4层人工目检(各30秒)，熔断机制保护写作节奏。 |
| 长篇失控 | L1冻结摘要——每章300-500字，写完即冻，永不回改。+ 6个追踪文件维护所有伏笔/角色/支线状态。 |

## 安装

```bash
git clone https://github.com/[username]/agent-novel.git
cd agent-novel
bash install.sh
```

`install.sh` 做了两件事：
1. 把 `SKILL.md` 复制到 `~/.claude/skills/agent-novel/`
2. 提示你把 `quality-check.sh` 复制到你的小说项目里

不需要 pip install。不需要 npm install。不需要 Docker。零运行时依赖。

## 使用

在 Claude Code 对话中输入：

```
写第79章
```

Skill 自动加载。Claude Code 会：
1. 自动读取你的 MASTER_SETTING（人物宪法）和 current_state（世界快照）
2. 按 6 条硬性规则写出正文
3. 运行 quality-check.sh 自动质检
4. 执行 3 项人工目检（命名/桥段/物理自洽）
5. 更新所有追踪文件

你不需要记住任何规则——skill 已经包含了所有规则。

## 6 条硬性规则

### 1. 抽象情绪词 0 容忍
感到、觉得、知道、认为、明白 — 禁止出现在叙述中。
不告诉读者"他害怕"——让他看到手抖、听到呼吸变快、感受到后背贴墙的凉意。

### 2. "不是X是Y"句式 ≤ 2 次/章
偶尔用一次有冲击力（"不是丧尸。是人。"）。用多了就是作者在说教。

### 3. 同一角色连续说话 ≤ 3 句
超过 3 句，插入动作/环境/沉默。防止对话变成"漂浮的台词"。

### 4. 段落 ≤ 250 字
250 字 ≈ 手机屏 10 行。超过这个长度，读者开始跳读。这是生理事实，不是文学判断。

### 5. 同一句式不连续复用
"他做了A。他做了B。他做了C。"——这不是风格，是模板 bug。

### 6. 不写角色解释他人心理状态
没有人能读心。角色当然也不能。让角色用观察和猜测替代"知道"。

> 每条规则都来自真实踩坑。详见：[docs/6-rules.md](docs/6-rules.md)

## 5 层质检

```
写完整章
  │
  ├→ 4a. quality-check.sh — 7项自动扫描，2秒
  │      ——数量 / "不是是"句式 / 抽象情绪词 / 字数 /
  │      一句话段落数 / 最长段落 / 连续逗号无句号
  │      FAIL → 修改 → 重跑（最多3轮）
  │
  ├→ 4b. 命名一致性 (30秒) — 秦队→秦队长？祺→祁？
  ├→ 4c. 桥段重复 (30秒) — 最近20章同类桥段去重
  └→ 4d. 物理/数字/元叙事 (30秒) — 5座车≠8人
```

> 详见：[docs/qc-system.md](docs/qc-system.md)

## L1 冻结摘要

长篇连载最致命的问题：写到后面忘了前面。

**L1 冻结摘要**：每章写完后，产出 300-500 字的摘要。写定后**永不修改**。如果后续章节发现摘要中的事实错误——不在摘要里改，在 `continuity-issues.md` 里追加修正记录。

这保证了：
- 任何时候回头看摘要，看到的是"当时的理解"
- 不会因为后来的剧情调整而"改写历史"
- 修正记录本身就是连载的成长档案

## 爽点梯级

### 战斗
- L1 被碾压 — 压倒性劣势，用计谋或代价勉强脱离
- L2 险胜 — 实力接近，靠某个细节翻盘，主角付出可见代价
- L3 智取 — 以弱胜强但合情合理，读者能复盘每一步
- L4 碾压 — 主角优势明显，必须有铺垫（非凭空变强）
- L5 不战而胜 — 通过之前的布局让他人代劳，或对方主动退让

### 情感
- L1 被理解 — 有人说出主角自己说不出口的话
- L2 和解 — 破裂关系的第一次重建信号
- L3 回应 — 长线情感的明确正面反馈
- L4 清算 — 压抑关系的终极爆发或终结

## 6 型章末钩子

信息炸弹 / 悬念断裂 / 情绪共振 / 反差沉默 / 动作戛然 / 命运转折

写新章时，自动避开前 3 章已用的钩子类型。防止读者对同一种钩子产生抗药性。

## 适用场景与不适用场景

### 适用
- 末世/公路/多角色/长篇连载
- 8 人以下主角团
- 章节长度 1800-5000 字
- 需要维护伏笔/支线/角色弧光的复杂叙事

### 不适用
- 短篇小说（< 10 章）— 追踪系统过重
- 单一角色视角 — 交互矩阵闲置
- 诗歌/散文 — QC 规则为叙事散文设计

## 设计原则

1. **最小改动拿最大收益**。不建维护地狱。
2. **每一处自动化都有明确的人和工具分工**。AI 负责写，QC 脚本负责检查，你负责裁决。
3. **规则服务于文字，不是反过来**。如果一条规则让文字更生硬——无视它。
