# 6 步流水线详解

> Agent Novel Skill 的完整写作流程，每一步都有明确的人和 AI 分工。

## 流水线总览

```
写第X章
  │
  ├→ 1. 准备（加载宪法+世界快照+前3章语感）
  │
  ├→ 2. 规划（单章3问+钩子类型+伏笔操作）
  │
  ├→ 3. 写作（1800-5000字·6条硬性规则）
  │
  ├→ 4. 质检（4a脚本扫描+4b命名+4c桥段+4d物理/数字/元叙事）
  │
  ├→ 5. 图片（可选）
  │
  └→ 6. 更新（6个追踪文件按序写入）
```

## 步骤 1：准备

### 必须加载（每次都读）

| 文件 | 内容 | 原因 |
|------|------|------|
| `constitution/MASTER_SETTING.md` | 人物宪法+世界观+交互矩阵 | 确保角色行为一致 |
| `tracking/current_state.md` | 世界快照（位置/物资/威胁/载具容量） | 防止物理穿帮 |
| 前 3 章正文首尾各 100 字 | 语感样本 | 保持叙事节奏一致 |

### 按需加载（仅当本章涉及时才读）

- 出场角色弧光状态 → `tracking/character_arcs.md`
- 感情线推进 → `tracking/emotional_arcs.md`
- 伏笔操作 → `tracking/pending_hooks.md`
- 支线平衡 → `tracking/subplot_board.md`
- 卷级弧光规划 → `arc_maps/arc_map_卷X.md`

### 上下文裁剪规则

当上下文接近 15k token 时，按序裁剪：
1. 写作指南（guides/ 下文件）
2. 非出场角色弧光
3. 关键词检索结果

**绝不裁剪**：MASTER_SETTING / current_state / 活跃伏笔

## 步骤 2：规划

回答三个问题：
1. **本章核心情绪？** — 用一个词定义（紧张/温暖/沉默/愤怒/释然）
2. **情感锚点角色？** — 谁承载本章的情感重量
3. **哪些可以不写？** — 删掉比添加更难，但更重要

然后：
- 选钩子类型（避开前 3 章已用类型）
- 确认本章伏笔操作（≥1 条：推进/回收/新埋）

## 步骤 3：写作

在 6 条硬性规则约束下完成 1800-5000 字正文。

6 条规则：
1. 抽象情绪词 0 容忍
2. "不是X是Y" ≤ 2 次/章
3. 同一角色连续说话 ≤ 3 句
4. 段落 ≤ 250 字
5. 同一句式不连续复用
6. 不写角色解释他人心理状态

## 步骤 4：质检（4 层 + 熔断）

### 4a. quality-check.sh（自动）

7 项自动化检查：
- ——数量（20-35）
- "不是/是"句式（≤2）
- 抽象情绪词（0）
- 正文字数（1500-5000）
- 一句话段落（≥5）
- 最长段落（≤250字）
- 连续逗号无句号（0）

不达标 → 修改 → 最多 3 轮。第 3 轮仍未通过 → 标记 [QC-FAIL] → 人工裁决。

### 4b. 命名一致性目检（30 秒）

扫本章主要人名/地名，确认无意外变体。
如"秦队"写成"秦队长"且非有意为之。

### 4c. 桥段重复目检（30 秒）

对照 chapter_summaries.md 最近 20 章的 [桥段类型] 标签，
确认本章无显性重复。

### 4d. 物理/数字/元叙事自洽（30 秒）

- **数字自洽**：叙述中声称的数量与实际是否一致（"四个字"实际五个字？）
- **物理约束**：载具容量/空间/时间距离是否合理（5 座车坐不了 8 个人？）
- **元叙事污染**：正文/对话是否出现"第X章""前文提到""如前所述"

## 步骤 5：图片（可选）

```bash
node scripts/convert-to-images.js X
```

## 步骤 6：更新追踪文件（按序写入，非并行）

1. `chapter_summaries.md` — 追加 L1 冻结摘要（300-500 字，写定冻存）+ 钩子类型 + [桥段类型]
2. `current_state.md` — 更新角色位置/团队状态/物资/威胁/载具空间
3. `pending_hooks.md` — 伏笔推进/回收/新埋
4. `subplot_board.md` — Strand Weave 日志 + 断档计数
5. `character_arcs.md` — 仅当弧光状态变化时更新
6. `emotional_arcs.md` — 仅当感情线/爽点变化时更新

## 每 5 章或卷末：健康度检查

对照 HARD_INVARIANTS.md 执行 4 项检查：
- HARD-001：可读性底线
- HARD-002：承诺违背检测
- HARD-003：节奏灾难检测
- HARD-004：冲突真空检测

结果追加到 logs/health-check-log.md。
